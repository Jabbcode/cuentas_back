## Why this matters

Tres problemas en la generación automática y alertas:

1. **Gastos fijos con `dueDay` 29–31 nunca se generan en meses cortos**: el cron filtra
   `dueDay: todayDay` con igualdad exacta; en un mes de 30 días, `dueDay=31` jamás
   coincide → el gasto fijo con autoGenerate nunca crea su transacción ese mes.
2. **El cron de las 9 AM y el mensual no tienen try/catch** (el de las 7 AM sí): una
   excepción dentro del callback async produce un unhandled rejection — en Node ≥15 eso
   tumba el proceso entero del servidor.
3. **N+1 + fallo en cascada en autoGenerate**: una query `findFirst` por cada gasto
   fijo, y si `createTransaction` lanza para uno, el loop entero aborta y los gastos
   restantes de ese día no se generan.

## Steps

### Step 1: try/catch en los jobs de 9 AM y mensual

En `src/lib/cron.ts`, envolver el cuerpo COMPLETO del callback de `'0 9 * * *'` en
`try { ... } catch (err) { console.error('[cron:debt-due]', err instanceof Error ? err.message : err); }`
y el del `'0 8 1 * *'` igual con etiqueta `[cron:monthly-email]`. Además, en el job de
las 9 AM, envolver el `createNotification` del loop en try/catch por-item para que un
fallo no corte las notificaciones de los demás usuarios.

**Verify**: `npx tsc --noEmit` → exit 0

### Step 2: Generar también en el último día del mes los dueDay desbordados

En `autoGenerateFixedExpenseTransactions`, reemplazar el filtro de igualdad exacta:

```ts
const todayDay = today.getDate();
const lastDayOfMonth = new Date(today.getFullYear(), today.getMonth() + 1, 0).getDate();
const isLastDay = todayDay === lastDayOfMonth;

const fixedExpenses = await prisma.fixedExpense.findMany({
  where: {
    isActive: true,
    autoGenerate: true,
    dueDay: isLastDay ? { gte: todayDay } : todayDay,
  },
  include: { user: { select: { id: true } } },
});
```

Así, el 30 de junio se generan los de dueDay 30 Y 31; el 28 de febrero, los de 28–31.
El check de "ya existe transacción este mes" evita duplicados si el gasto ya se generó.

**Verify**: `npx tsc --noEmit` → exit 0

### Step 3: Check de existentes en batch + try/catch por item

En la misma función, sustituir el `findFirst` dentro del loop por una sola query antes
del loop:

```ts
const existingTxs = await prisma.transaction.findMany({
  where: {
    fixedExpenseId: { in: fixedExpenses.map((fe) => fe.id) },
    date: { gte: startOfMonth, lte: endOfMonth },
  },
  select: { fixedExpenseId: true },
});
const alreadyGenerated = new Set(existingTxs.map((t) => t.fixedExpenseId));

for (const fe of fixedExpenses) {
  if (alreadyGenerated.has(fe.id)) continue;
  try {
    await createTransaction({ /* mismo payload actual */ }, fe.userId);
    createdByUser[fe.userId] = (createdByUser[fe.userId] ?? 0) + 1;
  } catch (err) {
    console.error('[cron:auto-generate]', `fe=${fe.id}`, err instanceof Error ? err.message : err);
  }
}
```

Nota: se elimina el filtro `userId` del check (el `fixedExpenseId` ya es del usuario
correcto porque viene de su propio gasto fijo).

**Verify**: `npx tsc --noEmit` → exit 0
